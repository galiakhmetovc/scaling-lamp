# 2026-04-12 — Telegram project capture looked like the agent forgot everything

## Формулировка с точки зрения пользователя

Для пользователя это выглядело так:

- агент только что нашёл и выполнил процедуру обновления шаблона Astra
- потом по просьбе "оформи как проект" создал пустую структуру
- после этого начал заново спрашивать, какой шаблон и что вообще делалось

То есть по пользовательскому восприятию агент **забыл всё**.

## Что видно в сыром trace

В trace ещё был недавний контекст про Astra:

- `/home/administrator/teamD/var/llm-traces/20260412T150244Z-run-8.json`
  - там есть найденный шаблон `tpl-astra-1.7-clean`
  - есть процедура обновления
  - есть итог выполнения с фактическим результатом

Потом в другом run агент сделал только scaffold проекта:

- `/home/administrator/teamD/var/llm-traces/20260412T152928Z-run-5.json`
  - создал `projects/template-update/...`
  - дальше опирался в основном на `memory.search`
  - recent work не был связан с project capture

## Реальная причина

Проблема не в отсутствии какого-либо воспоминания вообще, а в том, что:

1. команда "оформи как проект" не привязана к только что завершённому run
2. project capture не использует recent run replay / tool outputs / artifacts как primary source
3. агент слишком рано сваливается в `memory.search`
4. после этого начинает задавать повторные вопросы, хотя нужные факты были в недавнем run

## Правило

Для пользователя это должно считаться багом класса:

- "агент забыл только что сделанную работу"

Даже если часть данных ещё существует в trace или continuity, поведение считается неверным, если оператору приходится повторно объяснять только что завершённую задачу.

## Требование к будущей логике

Если пользователь просит:

- "оформи как проект"
- "сохрани это как проект"
- "запиши как кейс/процедуру"

то runtime должен в первую очередь использовать:

1. just-completed run
2. replay
3. tool events
4. artifacts
5. handoff / notes

и только потом `memory.search`.
